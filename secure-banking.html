Secure Banking Authentication

A .NET MAUI application that simulates how digital banking products combine multi-factor authentication, password hashing, and rule-based fraud detection—built with a service-oriented layout and clean separation of concerns.

Source on GitHub All projects
Solo C# .NET MAUI MFA / OTP Fraud rules Educational prototype

Context

Overview & goals

As banking moves online, protecting accounts from takeover and fraud is non-negotiable. This project is a deliberately scoped prototype (not production software) that demonstrates how layered controls— credentials, second factor, monitoring, and logging—fit together in a maintainable codebase.

The solution emphasises readable architecture: UI in MAUI, orchestration and rules in dedicated services, and documentation (including UML and a simple threat-oriented view) so design decisions stay visible to reviewers and future contributors.

Capabilities

What the app includes

Identity & access

  • User registration and login
  • OTP-based multi-factor authentication
  • Password hashing—no plain-text storage
  • Input validation and structured error handling

Operations & fraud

  • Transaction handling and history
  • Rule-based fraud detection (e.g. location, velocity, patterns)
  • Login location monitoring vs. expected “home” context
  • Fraud-alert logging and admin-oriented monitoring hooks

Structure

Architecture & services

The app follows a modular, MVC-inspired separation: MAUI pages and shell for the presentation layer, with service classes encapsulating domain behaviour. Core examples from the repository:

Key components

  • AuthService — authentication flow, credential verification, and OTP generation/validation.
  • FraudDetectionService — evaluates logins and transactions against configurable rules.
  • Data & persistence — database scripts and models aligned with the documented ERD.
  • Documentation folder — UML (use case, sequence), ERD, security notes, and engineering rationale.

Flow

Typical login path

  1. User signs in with username and password; credentials are checked through AuthService.
  2. An OTP is issued and must be confirmed before the session is treated as trusted.
  3. Client context (e.g. IP-derived location) is compared to stored expectations.
  4. FraudDetectionService scores the attempt; suspicious events are logged for review.
  5. After successful MFA, the user reaches the dashboard and transactional features.

Defence

Security highlights

Authentication

Passwords are hashed; MFA adds a second factor. Failed attempts can be tracked to resist brute-force style abuse in the simulated environment.

Fraud rules

Rules cover unfamiliar locations, repeated failures, and suspicious transaction shapes—mirroring how policy engines often start before ML is introduced.

Risks

Sample threat mitigations

Threat Mitigation (prototype)
Brute force Login attempt tracking and lockout-style behaviour in simulation
Credential theft Multi-factor authentication (OTP)
Session abuse Controlled post-MFA session flow
Fraudulent geography Location-aware login comparison
Suspicious transfers Rule-based transaction checks and alerts
