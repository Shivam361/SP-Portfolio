Studio Flow

A multi-tenant SaaS platform that automates graphic design studio operations end-to-end — from client intake and task assignment through revision cycles to final delivery. Built with Next.js 14, TypeScript, PostgreSQL, Redis, and a distributed BullMQ job system.

Context

Overview & goals

Graphic design studios run on shared drives, email threads, and WhatsApp groups — a setup that works until it doesn't. Missed briefs, unclear revision rounds, and no audit trail create friction that scales poorly as a studio grows. Studio Flow addresses this with a structured, multi-tenant operations platform akin to Jira or Perforce but purpose-built for creative workflows.

The platform handles the full project lifecycle: automated brief intake via IMAP email ingestion, Kanban-style task management, formal revision cycles with client sign-off, LAN-based asset handling with OneDrive cloud mirroring, multi-channel notifications (email & WhatsApp), and a comprehensive analytics layer for studio profitability and designer performance. Each studio is a fully isolated tenant — separate data, subdomain routing, and encrypted credential storage.

Role-based dashboards

What each role can do

Admin

  • User management with role assignment (Admin, Manager, Designer, Client)
  • Brief templates — reusable library for different deliverable types
  • LAN folder configuration (incoming / completed / archive paths)
  • IMAP / SMTP / Twilio credentials setup (AES-256-GCM encrypted at rest)
  • Real-time health monitoring — web, workers, email watcher, LAN watcher
  • Security audit log — all actions with IP, timestamp, and user context

Studio Manager

  • Kanban pipeline view — drag projects through Brief → Assigned → In Progress → Review → Delivered
  • Blueprint-based project creation with pre-populated task lists
  • Team workload management and deadline estimation
  • One-click revision round spawning with automated task creation
  • Token-gated client approval workflow with digital sign-off
  • OneDrive cloud sync — automatically mirrors completed deliverables via Microsoft Graph API
  • Performance analytics — on-time rates, revision cycles, profitability metrics

Designer

  • Task dashboard with assigned work, deadlines, and revision context
  • Full revision history and feedback per deliverable
  • Asset upload via web interface (Images, Video, Documents, Source Files, etc.)
  • Personal performance stats — on-time rate, average revision rounds, hours per deliverable type

Client

  • Time-limited token portal — no account required

  • Pixel-level visual annotation on submitted assets
  • Formal digital sign-off for B2B compliance and audit trail

System design

Architecture & multi-tenancy

Tenant isolation is enforced end-to-end. Middleware extracts the studio from the Host header on every request, resolves the tenant from a Redis-cached studio lookup, and injects it into the request context. Every database query goes through a repository pattern layer that mandates a studioId scope — making cross-tenant data leakage structurally impossible rather than a runtime concern.

Runtime processes

  • Web server — Next.js App Router with RSC + client components, serving all dashboards and APIs
  • BullMQ workers — 8 independent queues running as a separate Node.js process
  • IMAP watcher — polls the studio inbox, parses incoming emails into project briefs
  • LAN watcher — chokidar monitors the shared drive; attaches completed files to tasks automatically

Data layer

  • PostgreSQL with Prisma 7 ORM — 20+ models, soft deletes throughout
  • Repository pattern — strict data-access abstraction across all 50+ API endpoints
  • BlueprintTask model — first-class Prisma entity that spawns task records on project creation
  • StudioSettings — per-tenant encrypted SMTP / IMAP / Twilio credentials using AES-256-GCM

Async processing

Background job system

All heavy or time-sensitive operations are offloaded to a BullMQ queue system backed by Redis. Failed jobs are retried up to three times with exponential back-off; after exhausting retries they land in a Dead Letter Queue and trigger an email alert to the studio admin — ensuring no notification or sync operation silently disappears.

Queue inventory

  • Notification queue — dispatches email and WhatsApp messages for 14+ event types (brief submitted, task assigned, revision requested, approval received, deadline approaching, etc.)
  • Performance queue — recalculates designer stats (on-time rate, revision cycles, hours per type) after each task state change
  • OneDrive queue — uploads completed deliverables to the studio's OneDrive App Folder via Microsoft Graph API
  • Asset processing queue — generates Sharp image thumbnails and proxy files for in-browser preview
  • Token rotation queue — expires and regenerates time-limited client portal access tokens
  • DLQ alert queue — fires email to admin when any job exhausts its retry budget
